Product Requirements Document: Grok - AI Transcript Analysis & Structuring

1. Overview

Problem: While tools like the original Recall can generate transcripts from audio, there's still a significant manual effort required to extract specific, structured information, summaries, or insights from that raw text output. Users need a way to automatically process transcripts according to their specific needs, generating consistent, machine-readable data.

Solution: Grok will be a standalone application/module focused specifically on the AI-powered analysis of text transcripts. Users will provide a transcript text as input. They can then define or select "Analysis Profiles" containing instructions for an AI (initially Xai's Grok API) on what information to extract or generate. Grok will process the transcript against selected profiles sequentially and output structured JSON data, with the AI itself helping to define and refine the output schema for each profile.

Target Audience: Users who have text transcripts (potentially generated by Recall or other means) and need to perform customized, automated analysis to extract structured data, summaries, or insights. This includes the original developer, analysts, researchers, and anyone needing to convert unstructured transcript text into structured JSON based on specific criteria.

Value:

Automates the extraction of structured information (JSON) from unstructured transcripts.

Allows deep customization of analysis via user-defined profiles.

Leverages AI (Xai Grok) not just for analysis, but also for defining the structure (schema) of the output.

Provides flexibility in how analysis is performed (saved profiles vs. temporary prompts).

Generates machine-readable JSON output, ideal for downstream processing or database storage.

Separates the complex analysis logic from the initial transcription task (handled by Recall).

2. Core Features

Transcript Input:

What: Allows the user to input the text transcript to be analyzed (e.g., by pasting into a text area or potentially loading from a file).

Why: Provides the raw material for the AI analysis.

How: UI element (large text box or file selection) for providing the transcript content.

AI-Powered Transcript Analysis (via Xai Grok):

What: Processes the input transcript text using the Xai Grok LLM based on user-defined instructions (Analysis Profiles or a temporary prompt) to extract specific information and generate a structured JSON output. Runs analysis sequentially for each selected profile.

Why: The core function – transforming unstructured text into structured JSON based on user needs using a specific powerful LLM. Sequential processing ensures distinct results per profile.

How: Grok sends the transcript text and formatted instructions (derived from profiles/prompts) to the configured Xai Grok API endpoint. It receives the generated JSON output from the API for each profile processed.

Configurable Analysis Profiles:

What: System for users to create, save, name, manage, and select persistent analysis instructions ("profiles") via the UI. Each profile contains the prompt/instructions for the AI and an associated AI-generated/user-refined JSON schema definition.

Why: Enables reusable, tailored analysis tasks for different types of transcripts or information needs. Stores the intended structure alongside the instructions.

How: UI section for managing profiles (Create, List, Edit, Delete). Each profile stores a Name, Instructions (prompt), and Schema Definition (likely stored as JSON schema).

Flexible Analysis Execution:

What: Allows users to initiate analysis by:

Typing a one-off analysis instruction directly into a dedicated input area ("temporary prompt").

Selecting one or more saved Analysis Profiles (e.g., via checkboxes).

Why: Provides flexibility for both quick, ad-hoc analyses and standardized, repeatable tasks using saved profiles.

How: UI includes a text area for temporary prompts and a list/checkbox area for saved profiles. A "Process" button triggers analysis using either the temporary prompt or the selected saved profile(s). If multiple profiles are selected, the backend processes them one by one against the same input transcript.

AI-Driven Schema Generation & Editing:

What: Functionality integrated with Analysis Profile creation/editing that allows the AI (Xai Grok) to define and refine the JSON output schema.

Generate Schema: A button within the profile editor triggers the AI to suggest a JSON schema based on the profile's instructions.

Edit Schema (NLP Prompt): After viewing an output, users can provide natural language instructions (e.g., "add a field for 'priority'", "remove the 'notes' field") to modify the schema definition associated with the profile. This action only updates the schema, not the data itself immediately.

Regenerate Schema: Option to discard the current schema and have the AI generate it again based on the profile instructions.

Why: Leverages AI to assist in defining the output structure, making it easier for users to get desired JSON formats. Allows iterative refinement of the schema based on results.

How: Specific buttons/actions in the profile editor UI trigger backend calls to the Xai Grok API with specific meta-prompts focused on schema generation or modification based on the profile's main instructions or user's edit request. The resulting schema definition is saved with the profile.

Output Display (Tabs, JSON/Chat Views):

What: Displays the analysis results. If multiple profiles were run, results are shown in separate tabs, one for each profile. Within each tab, the user can toggle between:

JSON View: Shows the raw, structured JSON output as returned by the AI.

Chat View: Presents the information from the JSON in a more human-readable format, potentially using Markdown formatting derived from the JSON content.

Why: Provides both the precise machine-readable output and a user-friendly interpretation. Handles results from multiple analyses cleanly.

How: UI tabs dynamically generated based on processed profiles. A toggle control within each tab switches between rendering the raw JSON (e.g., using a formatted code viewer) and parsing the JSON to display it in a styled (Markdown) view.

Reprocess with Updated Schema:

What: After editing a profile's schema, a function to re-run the analysis for that specific profile using the same input transcript but applying the newly defined schema.

Why: Allows users to apply schema changes to the current transcript without manually re-initiating the whole process.

How: A "Reprocess" button, likely near the output display or profile editor, triggers the analysis again for the current transcript using the selected profile's (now updated) instructions and schema definition.

Modular LLM Integration Backend (Xai Grok Focused):

What: The component communicating with the LLM API must be modular, but the primary initial implementation will specifically target the Xai Grok API.

Why: Explicitly uses the desired API while retaining the architectural flexibility to potentially support other models or local instances later if needed.

How: An LLM_Connector interface with a primary, concrete implementation for XaiGrokConnector. Configuration settings point to the Xai Grok endpoint and manage API keys securely.

3. User Experience

User Personas:

Primary: Technical User / Developer / Analyst – Comfortable providing text input and defining analysis tasks. Needs structured JSON output. Appreciates AI assistance in schema definition but wants control over refinement.

Key User Flows:

Create & Save Profile w/ Schema: User navigates to "Manage Profiles" -> Clicks "Create New" -> Enters Name & Instructions -> Clicks "Generate Schema" -> AI suggests schema, user accepts or refines (initially just accepts) -> Saves Profile.

Run Analysis with Saved Profile(s): User pastes transcript text into input area -> Selects one or more saved profiles via checkboxes -> Clicks "Process" -> Grok processes sequentially -> Output tabs appear -> User clicks tabs, toggles JSON/Chat views.

Run Analysis with Temporary Prompt: User pastes transcript text -> Types analysis instructions into the temporary prompt area -> Clicks "Process" -> Grok processes -> A single output tab appears -> User views JSON/Chat.

Refine Schema & Reprocess: User views output for a profile -> Notices schema needs adjustment -> Goes to Edit Profile -> Uses "Edit Schema" NLP input (e.g., "Add 'urgency' field") -> Saves schema change -> Clicks "Reprocess" on the output tab -> Output updates using the new schema.

UI/UX Considerations:

Clear separation between Transcript Input, Profile Management/Selection, Temporary Prompt, and Output Display areas.

Intuitive interface for creating/editing profiles, including the schema generation/editing steps.

Obvious mechanism for selecting multiple profiles (checkboxes).

Clear visual indication of processing status.

Well-organized tabbed output for multiple results.

Easy-to-use toggle between JSON and Chat views. JSON view should be well-formatted. Chat view needs intelligent rendering from the JSON structure.

Clear controls for schema editing and reprocessing.

4. Technical Architecture

System Components:

Frontend (UI): Desktop application framework or Web UI. Handles transcript input, profile management, process initiation, results display (tabs, JSON/Chat toggle), schema interaction.

Backend/Core Logic: Orchestrates the workflow, manages Analysis Profiles (including schemas), interacts with the AI Analysis Module.

AI Analysis Module: Core component. Contains logic for:

Loading profile instructions and schemas.

Formatting prompts for Xai Grok (for both analysis and schema generation/editing).

Handling sequential processing of multiple profiles.

Interfacing with the XaiGrokConnector.

Receiving and potentially validating/standardizing JSON responses.

XaiGrokConnector (Modular Interface Implementation): Specific implementation. Handles authentication and communication with the Xai Grok API endpoint(s). Needs robust error handling.

Profile/Schema Storage: Simple database (like SQLite) or file-based storage (e.g., JSON files) to persist Analysis Profiles and their associated schema definitions.

Data Models:

TranscriptInput: Raw text content provided by the user.

AnalysisProfile: ID, Name, Instructions (text prompt), SchemaDefinition (JSON schema format), TimestampCreated, TimestampModified.

AnalysisJob: ID, associated TranscriptInput, list of AnalysisProfileIDs to run or TemporaryInstructions, Status (Pending, Processing, Complete).

AnalysisResult: ID, associated AnalysisJobID, associated AnalysisProfileID (if applicable), TimestampCreated, OutputJSON (text/JSON blob), SchemaUsed (reference to the schema definition used), Status (Success, Failed).

APIs and Integrations:

Xai Grok API: Primary external dependency. Requires secure handling of API keys and robust interaction logic (requests, responses, error handling).

Infrastructure Requirements:

Execution Environment: Local machine (Python environment, etc.) or potentially web server if built as a web app.

Xai Grok Access: Internet connection and valid API credentials.

(Focus on scope and features per phase, not timelines)

Phase 1: MVP - Core Analysis Flow (Single Profile, Basic Schema)

Goal: Establish the fundamental workflow: input transcript, run one saved profile via Xai Grok, display raw JSON and basic Chat view output. AI determines schema implicitly during analysis.

Requirements:

Implement XaiGrokConnector for basic API calls. Secure API key handling.

Basic storage for AnalysisProfile (Name, Instructions - no explicit schema stored yet). UI to Create/List/Select one profile.

UI for transcript text input.

Backend logic to take input text + selected profile instructions, call Xai Grok via connector, receive JSON output.

UI Output area: Display raw JSON (JSON View). Implement a very basic Chat View (e.g., just pretty-printing the JSON).

"Process" button to trigger the flow. Minimal error handling.

Phase 2: Multi-Profile Processing & Explicit Schema Generation

Goal: Enable processing multiple profiles sequentially, enhance output display, and add explicit AI schema generation.

Requirements:

UI to allow selecting multiple saved profiles (checkboxes).

Backend logic to iterate through selected profiles, calling the analysis for each sequentially.

UI Output area: Implement tabbed display, one tab per processed profile result.

Implement the "Generate Schema" button in the profile editor: Calls AI to get schema, store SchemaDefinition with the AnalysisProfile.

Modify analysis call to instruct the AI to use the stored schema when generating JSON.

Implement the temporary prompt input area and allow processing using it (outputs to a single tab).

Phase 3: Schema Editing, Reprocessing & Enhanced Chat View

Goal: Implement advanced schema refinement capabilities and improve the user-friendliness of the output.

Requirements:

Implement "Edit Schema" functionality using NLP prompts (UI input + backend call to AI for schema modification). Update stored SchemaDefinition.

Implement "Regenerate Schema" button.

Implement the "Reprocess" button on the output tab to re-run analysis for that profile with the current transcript and updated schema.

Enhance Chat View rendering: Implement logic to parse the output JSON and display it using Markdown for better readability (lists, bolding, etc.).

Future Enhancements:

(Grok Internal) Database storage/caching of results.

(Grok Internal) Advanced JSON validation against schema.

(Integration) Develop mechanisms to link Grok analysis runs/results back to the original Recall audio/transcripts (e.g., passing IDs, using a shared database).

(Integration) Develop mechanisms for Recall to automatically send completed transcripts to Grok for analysis.

(Downstream) Features/modules to process the JSON outputs generated by Grok (as originally planned for Recall's next steps after analysis).

7. Risks and Mitigations

Xai Grok API Dependency: Availability, cost, performance, rate limits, or changes to the API. Potential for vendor lock-in if modularity isn't maintained.

Mitigation: Design LLM_Connector modularly from the start, even if only implementing the Xai Grok version initially. Monitor API status and costs. Implement appropriate error handling and retry logic. Have placeholder plans for alternative models if needed.

AI Schema Generation/Editing Reliability: The AI might misunderstand instructions for schema generation or editing, leading to incorrect or unusable schemas/outputs. Getting consistent JSON structure can be challenging.

Mitigation: Use strong meta-prompts for schema tasks. Allow manual JSON schema editing as a fallback. Start with simpler schema requirements. Extensive testing with various instruction types. Provide clear feedback to the user on schema changes.

UI Complexity: Managing profiles, schemas, inputs, outputs (tabs, views), and reprocessing requires a potentially complex UI.

Mitigation: Phased UI development. Focus on clarity and intuitive workflows. Get user feedback early and often (even if self-feedback). Keep UI elements grouped logically.

Cost Management: Xai Grok API calls will incur costs, especially schema generation/editing and reprocessing.

Mitigation: Make processing steps explicit user actions (buttons). Be mindful of potentially chatty interactions for schema editing. Consider cost warnings or usage tracking if it becomes significant.

Deferred Integration Complexity: Integrating Grok back with Recall later might present unforeseen challenges.

Mitigation: Keep potential integration points in mind during Grok's design (e.g., use of common identifiers if possible). Document Grok's API or data formats clearly for future integration.
